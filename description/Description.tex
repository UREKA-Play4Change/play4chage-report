\section{Descrição do Projecto}
%% ============================================================

\subsection{Contexto e Motivação}

A transição para cidades inteligentes enfrenta três obstáculos interdependentes: infraestruturas tecnológicas, constrangimentos financeiros e de governança, e --- o foco deste trabalho --- \textbf{o elemento humano}: as competências, a motivação e a participação dos cidadãos. A Aliança Universitária \textbf{U!REKA} tem como missão acelerar precisamente esta transição.

O \textit{Play4Change} propõe uma plataforma de \textbf{aprendizagem adaptativa e gamificada para educação cívica urbana}, desenvolvida no âmbito da missão U!REKA e alinhada com os Objetivos de Desenvolvimento Sustentável das Nações Unidas (ODS~4, 11 e~13). A plataforma assenta em três pilares:

\begin{enumerate}
  \item \textbf{Engagement --- Desafios diários gamificados}: transformam a aprendizagem cívica num hábito diário motivante, através do qual os cidadãos desenvolvem micro-competências progressivas;

  \item \textbf{Personalização --- Desafios adaptativos por IA}: um agente de inteligência artificial (Mistral AI) adapta os percursos ao perfil e às dificuldades de cada cidadão. Quando gera um percurso alternativo, este é indexado em \textit{pgvector} e reutilizado automaticamente para perfis análogos, reduzindo a dependência de inferência em tempo real;

  \item \textbf{Reconhecimento --- Badges e micro-competências}: cada etapa de progressão é certificada por um \textit{badge}, tornando cada passo no percurso visível e contabilizável.
\end{enumerate}

\subsection{Arquitectura do Sistema}

O sistema foi concebido segundo os princípios de \textbf{Arquitetura Hexagonal} (Ports \& Adapters), em cinco contextos delimitados (DDD):

\begin{center}
\begin{tabular}{lp{9cm}}
\toprule
\textbf{Contexto} & \textbf{Responsabilidade} \\
\midrule
\textit{Identity} & Registo, autenticação (magic link), gestão de perfil \\
\textit{Topic}    & Criação de tópicos via URL/PDF, geração de tarefas por IA (pipeline assíncrono) \\
\textit{Enrollment} & Inscrição, entrega diária de tarefas, progressão por dia com timezone \\
\textit{Struggle} & Sessões de remediação adaptativa quando o utilizador falha \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Comportamento sem Chaves Externas}

O sistema foi desenhado para funcionar em modo degradado quando as chaves externas não estão disponíveis:

\begin{itemize}
  \item \textbf{Sem \env{MISTRAL\_API\_KEY}}: a geração de tarefas usa um adaptador stub (\texttt{NoOpTaskGenerationAdapter}) — os tópicos são criados mas as tarefas ficam vazias. Para ver IA em acção, a chave deve estar configurada no \file{.env}.

  \item \textbf{Sem \env{RESEND\_API\_KEY}}: os e-mails de magic link são impressos nos \textbf{logs do servidor} em vez de serem enviados. O júri pode obter o link directamente nos logs com:
  \begin{lstlisting}[style=bash]
  docker compose logs server | grep "magic-link"
  \end{lstlisting}

\end{itemize}

\section{Instalação e Arranque}
%% ============================================================

\subsection{Passo 1 — Clonar o Repositório Principal}

\begin{lstlisting}[style=bash]
git clone https://github.com/UREKA-Play4Change/play4change-core.git
cd play4change-core
\end{lstlisting}

\subsection{Passo 2 — Verificar o Ficheiro \texttt{.env}}

O ficheiro \file{.env} já existe na raiz do repositório. Verificar que está presente:

\begin{lstlisting}[style=bash]
ls -la .env
cat .env   # confirmar que os valores estao preenchidos
\end{lstlisting}

\subsection{Passo 3 — Arrancar todos os Serviços}

\begin{lstlisting}[style=bash]
docker compose up --build
\end{lstlisting}

Na primeira execução, o Docker irá:
\begin{enumerate}
  \item Descarregar as imagens base (pgvector, MinIO, Redis, Nginx, Prometheus, Grafana);
  \item Compilar o servidor Spring Boot (demora 2--4 minutos);
  \item Executar as 23 migrações Flyway automaticamente;
  \item Iniciar todos os serviços com health checks.
\end{enumerate}

O sistema está pronto quando se vir no terminal:

\begin{lstlisting}
play4change-server  | Started Application in X.XXX seconds
\end{lstlisting}

Para correr em background:
\begin{lstlisting}[style=bash]
docker compose up --build -d
\end{lstlisting}

\subsection{Passo 4 — Confirmar que os Serviços estão Activos}

\begin{lstlisting}[style=bash]

# Todos os servicos devem estar em estado "running" ou "healthy"
\end{lstlisting}

\service{minio}      & 9000, 9001 & healthy \\

\section{Guia de Verificação para o Júri}
%% ============================================================

Esta secção descreve o percurso completo que o júri pode seguir para verificar que o sistema funciona correctamente, da autenticação à entrega adaptativa de tarefas.

\subsection{Verificação 1 — Health Check da API}

\begin{lstlisting}[style=bash]
curl http://localhost:9090/actuator/health
# Resposta esperada:
# {"status":"UP","components":{"db":{"status":"UP"},...}}
\end{lstlisting}

Ou abrir no browser: \href{http://localhost:9090/actuator/health}{\texttt{http://localhost:9090/actuator/health}}

\subsection{Verificação 2 — Documentação Swagger da API}

Abrir no browser:

\begin{mdframed}[style=infobox]
\centering
\href{http://localhost:8080/swagger-ui.html}{\large\texttt{http://localhost:8080/swagger-ui.html}}
\end{mdframed}

Todos os endpoints estão documentados e podem ser testados directamente na interface Swagger.

\subsection{Verificação 3 — Criar uma Conta (Magic Link)}

\begin{enumerate}
  \item Aceder ao painel de administração web em: \href{http://localhost:80}{\texttt{http://localhost:80}}
  \item No ecrã de login, introduzir um endereço de e-mail (qualquer e-mail, real ou fictício).
  \item Clicar em \textbf{``Enviar Magic Link''}.
  \item Como o ambiente é local sem chave Resend, obter o link nos logs do servidor:
  \begin{lstlisting}[style=bash]
  docker compose logs server | grep -i "magic\|verify\|token"
  \end{lstlisting}
  \item Copiar o URL de verificação do log e abrir no browser. A sessão é autenticada automaticamente.
\end{enumerate}

\begin{mdframed}[style=warningbox]
\textbf{Alternativa via API directamente (sem frontend):}
\begin{lstlisting}[style=bash]
# 1. Solicitar magic link
curl -X POST http://localhost:8080/auth/magic-link \
  -H "Content-Type: application/json" \
  -d '{"email": "jury@example.com"}'

# 2. Ver o token nos logs
docker compose logs server | grep "verify"

# 3. Verificar o token (substituir TOKEN pelo valor obtido)
curl "http://localhost:8080/auth/verify?token=TOKEN"
# Resposta: {"accessToken":"...","refreshToken":"..."}
\end{lstlisting}
\end{mdframed}

\subsection{Verificação 4 — Promover Conta a Administrador}
\label{sec:promote}

Após criar a conta com o e-mail \texttt{jury@example.com} (ou o e-mail escolhido), promover a administrador:

\begin{lstlisting}[style=bash]
# Na raiz do repositorio play4change-core
chmod +x scripts/promote-admin.sh
./scripts/promote-admin.sh jury@example.com
\end{lstlisting}

\textbf{Saída esperada:}
\begin{lstlisting}
OK: 'jury@example.com' promoted to ADMIN.
They must log out and log in again for the new role to take effect.
\end{lstlisting}

Após a promoção, fazer logout e novo login para o JWT reflectir o papel de ADMIN.

\subsection{Verificação 6 — Criar um Tópico de Aprendizagem via IA}

Este passo requer a chave \env{MISTRAL\_API\_KEY} configurada no \file{.env}. Com a conta ADMIN autenticada:

\begin{mdframed}[style=infobox]
\textbf{Via Painel Web (recomendado):}
\begin{enumerate}
  \item Navegar para \texttt{http://localhost/admin/topics/new}
  \item Introduzir um URL de conteúdo cívico (ex.: página Wikipedia sobre reciclagem urbana)
  \item Seleccionar dificuldade, idioma e categoria
  \item Clicar em \textbf{``Criar Tópico''}
  \item Acompanhar o progresso em tempo real (SSE) na página de detalhes do tópico
\end{enumerate}
\end{mdframed}

\begin{mdframed}[style=warningbox]
\textbf{Via API (com token JWT de ADMIN):}
\begin{lstlisting}[style=bash]
curl -X POST http://localhost:8080/admin/topics \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ACCESS_TOKEN" \
  -d '{
    "sourceUrl": "https://pt.wikipedia.org/wiki/Reciclagem",
    "language": "pt",
    "durationDays": 7,
    "difficulty": "BEGINNER",
    "category": "ENVIRONMENT"
  }'
\end{lstlisting}
Acompanhar progresso: \cmd{GET /admin/topics/\{id\}/progress} (SSE stream)
\end{mdframed}

\subsection{Verificação 7 — Inscrição num Tópico e Tarefa Diária}

Com a conta (papel USER) e um tópico activo (\texttt{status: ACTIVE}):

\begin{lstlisting}[style=bash]
# Inscrever no topico (substituir TOPIC_ID e ACCESS_TOKEN)
curl -X POST http://localhost:8080/topics/TOPIC_ID/enroll \
  -H "Authorization: Bearer ACCESS_TOKEN"

# Obter a tarefa do dia
curl http://localhost:8080/tasks/today?topicId=TOPIC_ID \
  -H "Authorization: Bearer ACCESS_TOKEN"

# Submeter resposta (substituir ASSIGNMENT_ID e escolha A/B/C/D)
curl -X POST http://localhost:8080/tasks/ASSIGNMENT_ID/submit \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ACCESS_TOKEN" \
  -d '{"selectedOption": "A"}'
\end{lstlisting}

\section{Painel de Administração Web}
%% ============================================================

O painel web (\texttt{play4change-web}) está incluído na imagem Docker do Nginx em modo de produção. Não é necessário correr o servidor de desenvolvimento Node.js para aceder ao painel.

\begin{center}
\begin{tabularx}{\textwidth}{lX}
\toprule
\textbf{URL} & \textbf{Funcionalidade} \\
\midrule
\texttt{http://localhost/} & Página pública com missão e estatísticas em tempo real \\
\texttt{http://localhost/admin/login} & Login (magic link) \\
\texttt{http://localhost/admin/dashboard} & Dashboard com resumo de actividade \\
\texttt{http://localhost/admin/topics} & Listagem de todos os tópicos (com filtros de estado) \\
\texttt{http://localhost/admin/topics/new} & Criar tópico a partir de URL ou PDF \\
\texttt{http://localhost/admin/topics/\{id\}} & Detalhe, edição de tarefas, pré-requisitos \\
\texttt{http://localhost/admin/learning-paths} & Visualização do grafo de pré-requisitos (DAG) \\
\texttt{http://localhost/admin/users} & Gestão de utilizadores e promoção a ADMIN \\
\texttt{http://localhost/admin/reports} & Relatórios de tarefas sinalizadas pelos utilizadores \\
\texttt{http://localhost/admin/badges} & Badges de micro-competências por tópico \\
\bottomrule
\end{tabularx}
\end{center}

\begin{mdframed}[style=warningbox]
\textbf{Nota sobre autenticação:} O painel web requer conta com papel \textbf{ADMIN}. A primeira conta criada via magic link tem papel USER por defeito. Usar o script \cmd{./scripts/promote-admin.sh \textless{}email\textgreater{}} para promover a ADMIN (ver Secção~\ref{sec:promote}).
\end{mdframed}
