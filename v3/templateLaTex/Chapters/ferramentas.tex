\chapter{Ferramentas, Tecnologias e Metodologia}
\label{cha:ferramentas}

Este capítulo descreve o conjunto de tecnologias que constituem o ecossistema técnico do \emph{Play4Change}, a pipeline de integração e entrega contínua que garante a qualidade do código em cada alteração, e a forma como a inteligência artificial foi utilizada ao longo do próprio processo de desenvolvimento --- não apenas como funcionalidade da plataforma, mas como ferramenta de trabalho.

\subsection*{Servidor}

O servidor foi implementado em \textbf{Kotlin} com o \emph{framework} \textbf{Spring Boot 3.2}, seguindo uma arquitectura hexagonal (\emph{Ports \& Adapters})~\cite{cockburn2005hexagonal} que separa a lógica de negócio das suas dependências externas. A escolha do Kotlin em vez do Java tradicional permitiu aproveitar a expressividade da linguagem --- em particular as \emph{coroutines} para operações assíncronas, os tipos não-anuláveis para redução de erros de \emph{null pointer}, e o estilo funcional para as transformações de dados. O Spring Boot forneceu a base REST, a gestão de transações e o suporte à autenticação com JWT~\cite{rfc7519}.

A base de dados \textbf{PostgreSQL 16} foi estendida com a extensão \textbf{pgvector}, que adiciona suporte a vectores de \emph{embeddings} directamente no mesmo motor relacional. Esta co-localização elimina a necessidade de um sistema vectorial separado e mantém todas as garantias ACID para a camada de personalização. O \textbf{Redis} é utilizado para a cache de resultados de consultas frequentes e para controlo de ritmo das chamadas à API de IA. A integração com o modelo de linguagem \textbf{Mistral 7B} é feita através do \textbf{LangChain4j 0.36.2}~\cite{langchain4j}, implementando o padrão RAG~\cite{lewis2020rag} descrito no Capítulo~\ref{cha:enquadramento}.

\subsection*{Aplicação móvel}

A aplicação móvel foi desenvolvida com \textbf{Kotlin Multiplatform (KMP) 2.3} e \textbf{Compose Multiplatform 1.10}, o que permite partilhar a maior parte da lógica de negócio e da interface entre Android e iOS a partir de um único código-base. A navegação e a gestão de estado são tratadas pelo \textbf{Decompose 3.3}, que oferece uma abstracção de \emph{stack} de componentes compatível com ambas as plataformas e com testes unitários.

\subsection*{Painel de administração}

O painel web de administração é uma aplicação \textbf{React 18} em \textbf{TypeScript}, construída com \textbf{Vite} e \textbf{Tailwind CSS}. Permite criar e gerir tópicos, tarefas e conteúdos, monitorizar inscrições e visualizar dados de utilização. O \textbf{TanStack Query} é utilizado para a gestão de estado remoto, garantindo cache automática, revalidação e tratamento consistente de estados de carregamento e erro.

\subsection*{Infraestrutura de produção}

Todo o ambiente de produção é definido num único ficheiro \texttt{docker-compose.yml} com sete serviços: servidor Spring Boot, PostgreSQL, Redis, MinIO, nginx, Prometheus e Grafana. O \textbf{nginx} actua como \emph{reverse proxy} --- recebe os pedidos externos na porta 80, encaminha as chamadas à API (prefixadas em \texttt{/play4change-server/}) para o servidor na porta 8080, e serve o cliente React como conteúdo estático. O \textbf{Prometheus} recolhe métricas expostas pelo actuador Spring Boot a cada 15 segundos e o \textbf{Grafana} apresenta-as em três \emph{dashboards} pré-configurados: um \emph{dashboard} geral, um centrado na latência da IA e um que mostra o fluxo do aprendente em tempo real. \subsection*{Workflow 1 --- Integração Contínua}

O \emph{workflow} de CI é desencadeado em todos os \emph{commits} para a \emph{branch} \texttt{main} e em todos os \emph{pull requests} que a ela se destinam. Corre num agente \emph{macOS} (exigência do \emph{toolchain} Android) e executa quatro passos em sequência:

\begin{enumerate}
	\item \textbf{Gitleaks} --- varredura de segredos no histórico completo do repositório (\texttt{fetch-depth:~0}). Se forem detectadas credenciais, chaves de API ou outros segredos comprometidos no código, o \emph{workflow} falha imediatamente antes de qualquer compilação.
	\item \textbf{JDK 21 (Eclipse Temurin)} --- configuração do ambiente Java com cache automática das dependências Gradle entre execuções, reduzindo o tempo de compilação em execuções subsequentes.
	\item \textbf{Gradle build + testes} --- compila todos os módulos (servidor, móvel, \emph{common}) e executa os testes unitários e de integração com JUnit~5, MockK e SpringMockK. O passo exclui explicitamente o \emph{packaging} de \emph{release} para reduzir o tempo de execução em CI.
	\item \textbf{Detekt} --- análise estática do código Kotlin do servidor, verificando aderência às regras definidas em \texttt{detekt.yml}: complexidade ciclomática, tamanho de funções, dependências em concretos em vez de interfaces, e estilo de código.
\end{enumerate}

Apenas se todos os passos passarem é que um \emph{pull request} pode ser integrado na \emph{branch} principal. A Tabela~\ref{tab:ci-gates} resume os critérios de qualidade verificados em cada passo.

\subsection*{Workflow 2 --- Release do APK Android}

O \emph{workflow} de \emph{release} é desencadeado exclusivamente por \emph{tags} com prefixo \texttt{v} (por exemplo, \texttt{v1.2.3}). Este mecanismo garante que os binários de produção são sempre rastreáveis a uma versão explícita. O \emph{workflow}: restaura os ficheiros sensíveis (\emph{keystore} e \texttt{google-services.json}) a partir de segredos encriptados do repositório, nunca armazenados em texto simples; calcula automaticamente o \texttt{versionCode} Android a partir da \emph{tag} semântica, segundo a fórmula $\text{MAJOR} \times 10000 + \text{MINOR} \times 100 + \text{PATCH}$; compila e assina o APK de \emph{release} com a \emph{keystore} de produção; e cria automaticamente uma \emph{GitHub Release} com o \texttt{play4change.apk} anexado, tornando-o imediatamente disponível para distribuição sem qualquer passo manual.

\subsection*{Workflow 3 --- Verificação semanal de dependências (OWASP)}

O terceiro \emph{workflow} corre automaticamente todas as segundas-feiras às 06:00 UTC. Utiliza a ferramenta \textbf{OWASP Dependency Check} para analisar todos os JARs do servidor e dos módulos móveis em busca de vulnerabilidades conhecidas registadas na base de dados CVE (\emph{Common Vulnerabilities and Exposures}). O \emph{workflow} falha se forem encontradas vulnerabilidades com \emph{score} CVSS igual ou superior a 7 (``\emph{High}'' ou ``\emph{Critical}''), forçando uma resposta atempada. Os relatórios HTML gerados ficam disponíveis como artefactos no GitHub durante 30 dias. Esta verificação periódica é particularmente relevante para uma plataforma que usa um LLM externo e que processa dados de utilizadores --- onde uma dependência comprometida poderia ter impacto directo na privacidade dos dados ou na integridade dos conteúdos gerados.

\section{Metodologia de desenvolvimento}
\label{sec:metodologia-desenvolvimento}
% ─────────────────────────────────────────────────────────────────────────────

O desenvolvimento do \emph{Play4Change} seguiu um processo iterativo e incremental, próximo de uma metodologia ágil informal adequada a uma equipa de dimensão reduzida. O controlo de versões foi feito com \textbf{Git} e o repositório alojado no \textbf{GitHub}, com uma estratégia de \emph{feature branches}: cada nova funcionalidade ou correcção é desenvolvida numa \emph{branch} dedicada (por exemplo, \texttt{feature/jwt-auth} ou \texttt{bugfix/daily-reminders}), submetida como \emph{pull request} e sujeita a revisão de código antes de ser integrada na \emph{branch} \texttt{main}. Toda a integração é protegida pela pipeline de CI descrita na secção anterior.

A gestão de dependências é centralizada no ficheiro \texttt{gradle/libs.versions.toml}, o que garante consistência entre todos os módulos do projecto e facilita actualizações coordenadas. O \textbf{Flyway} é utilizado para a gestão de migrações da base de dados: cada alteração ao esquema SQL é registada como um ficheiro de migração versionado, aplicado de forma automática e ordenada aquando do arranque do servidor, garantindo que o estado da base de dados é sempre rastreável e reproduzível em qualquer ambiente.
